# 프론트 시작

## 사용 예시
```
# 기본 (설계문서 자동 로드, Stitch 없음)
/프론트시작

# Stitch 디자인 포함
/프론트시작 Stitch Project ID: abc123

# Stitch + 설계 승인 명시
/프론트시작 Stitch Project ID: abc123, 설계승인됨

# 특정 화면만 Stitch에서 가져오기
/프론트시작 Stitch Project ID: abc123, 화면: screen-001 screen-002

# 기존 코드 버그 수정
/프론트시작 [버그] 로그인 폼 제출 후 로딩 스피너 안 꺼지는 문제

# 기존 코드 고도화
/프론트시작 [고도화] 상품 목록 페이지 렌더링 성능 개선

# 기존 코드 기능 변경
/프론트시작 [변경] 결제 버튼 위치 하단 고정으로 변경 (고객 요청)
```

> 이 커맨드는 오케스트레이터의 GATE 2 승인 후에만 실행한다.
> "설계 승인" 또는 "코드 생성 시작해" 명령 없이 실행된 경우 GATE 2 승인 여부를 먼저 확인한다.

---

아래 순서로 진행한다.

## 1. 에이전트 하네스 로드
`agents/frontend.md` 파일을 읽고 프론트 에이전트 규칙을 완전히 숙지한다.

## 2. 입력 확인
```
$ARGUMENTS
```

입력이 없는 경우 아래를 자동으로 로드한다.
- `docs/설계/openapi.yaml` ← 계약서 원본
- `docs/설계/architecture.md` ← Clean Architecture 레이어 구조
- `docs/기획/` ← 유저 스토리 확인

## 3. Google Stitch 디자인 수신 (있는 경우)
입력에 Stitch Project ID 가 포함된 경우 Stitch MCP 를 통해 디자인을 가져온다.

```
① extract_design_context (Project ID)
   → .stitch/DESIGN.md 저장 (색상/타이포그래피/간격/컴포넌트 패턴)

② build_site (Project ID + 라우트 매핑)
   또는 get_screen_code (화면별 개별 추출)
   → 각 화면의 HTML/CSS 코드 추출

③ get_screen_image (주요 화면)
   → 스크린샷으로 시각 확인
```

Stitch Project ID 가 없는 경우 이 단계를 건너뛴다.
디자인 없이 구현하는 경우 오케스트레이터에게 확인 후 진행한다.

```
## Stitch 디자인 수신 완료 보고 (해당 시)
- DESIGN.md 저장 위치: .stitch/DESIGN.md
- 추출된 화면 수: {N}개
- 주요 디자인 토큰:
  - Primary Color: {색상}
  - Font: {폰트}
  - Spacing: {간격 시스템}
- 미확인 화면: {있을 경우}
```

## 4. 설계 문서 분석
- openapi.yaml 전체 엔드포인트 및 스키마 파악
- 프론트엔드 레이어 구조 확인
- 구현할 페이지 및 컴포넌트 목록 도출
- DESIGN.md 가 있는 경우 디자인 토큰 목록 파악

## 5. 기능 단위 분리 계획 수립
유저 스토리 기반으로 feature 브랜치 목록을 작성하고 오케스트레이터에게 보고한다.

```
## 프론트 기능 단위 분리 계획
| 순서 | 브랜치명 | 구현 내용 | Stitch 화면 | 의존성 |
|------|---------|---------|-----------|-------|
| 1 | feature/{명} | {내용} | {화면 ID or 없음} | 없음 |
| 2 | feature/{명} | {내용} | {화면 ID or 없음} | 1번 완료 후 |

진행 방법: 각 단위 완료 시 코드 리뷰 요청 후 develop 머지
승인 요청: 위 계획대로 진행할까요?
```

## 6. 기능 단위별 구현 (Git Flow 자동 반복)
오케스트레이터 확인 후 아래 사이클을 반복한다.

```
[각 기능 단위마다]
① feature 브랜치 생성
② 컴포넌트 설계 → 아키텍처 감시자 검토
③ Stitch HTML/CSS → Next.js + TailwindCSS 컴포넌트 변환 (DESIGN.md 참조)
④ 구현 + API 연동 (병렬)
⑤ 단위 테스트 + MSW 목 작성
⑥ ESLint / Prettier / tsc PASS
⑦ 코드 리뷰 요청 (오케스트레이터 응답 대기)
⑧ "머지해" 수신 → develop 머지
⑨ 다음 단위로 이동
```

## 7. 최종 품질 검사
전체 구현 완료 후 전체 코드 대상 최종 검사를 수행한다.
```
ESLint / Prettier / tsc --noEmit / any 0건 / console.log 0건
```

## 8. QA 자동 인계
품질 검사 완료 후 QA 에이전트에게 자동 인계한다.

```
## 프론트 구현 완료 보고
- 구현 완료 기능: {목록}
- Stitch 디자인 적용 화면: {N}개
- 테스트 통과: {N}건
- 코드 품질: 전체 PASS
- 미결 사항: {있을 경우}
→ /QA시작 자동 실행
```

# 프론트 수정 시작 (기존 코드)

$ARGUMENTS 에 아래 중 하나가 포함된 경우 기존 코드 수정 에이전트로 실행한다.
- `[버그]`, `[고도화]`, `[변경]` 키워드
- 특정 파일/컴포넌트 이름 언급

수정 에이전트 실행 시 `agents/frontend.md` 의 **기존 코드 수정 에이전트** 섹션을 따른다.

```
① 코드 분석 → ② 영향도 분석 → ③ 수정 계획 보고
→ 오케스트레이터 확인 → ④ 구현 → ⑤ 회귀 테스트 → ⑥ 코드 리뷰 → 머지
```
