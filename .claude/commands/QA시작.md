# QA 시작

## 사용 예시
```
# 기본 (자동 실행 - 프론트+백엔드 완료 후)
/QA시작

# 특정 기능만 QA
/QA시작 로그인 기능만 테스트

# 버그 수정 후 회귀 테스트
/QA시작 [회귀] BUG-012 수정 후 재테스트

# 테스트 서버 정보 포함 (시나리오 테스트 바로 시작)
/QA시작 서버: https://test.example.com / 관리자: admin@test.com:pw123 / 사용자: user@test.com:pw456

# 보안 테스트만 집중
/QA시작 [보안] 인증/인가 취약점 집중 검증
```

> 이 커맨드는 프론트 + 백엔드 에이전트 구현 완료 후 자동 실행되거나
> 오케스트레이터가 직접 호출할 수 있다.

---

아래 순서로 진행한다.

## 1. 에이전트 하네스 로드
`agents/qa.md` 파일을 읽고 QA 에이전트 규칙을 완전히 숙지한다.

## 2. 입력 확인
```
$ARGUMENTS
```

입력이 없는 경우 아래를 자동으로 로드한다.
- `docs/기획/` ← 유저 스토리 전체
- `docs/설계/openapi.yaml` ← API 스펙
- `docs/설계/security-design.md` ← 보안 요구사항
- 프론트/백엔드 구현 코드 경로

## 3. 테스트 계획 수립
유저 스토리와 openapi.yaml 기반으로 테스트 케이스 전체를 설계한다.

```
## 테스트 계획
- 유저 스토리 커버리지: {N}/{N}
- API 엔드포인트 커버리지: {N}/{N}
- 보안 테스트 케이스: {N}건
- 총 테스트 케이스: {N}건
```

## 4. 통합 테스트 + 보안 테스트 (병렬 자동 실행)

### API 통합 테스트
- 모든 엔드포인트 정상 응답 + 스키마 일치
- 에러 응답 구조 일치
- 인증 토큰 처리 (발급/갱신/만료)

### 보안 테스트
- 인증/인가 (미인증 401 / 권한없음 403 / 타인 리소스 차단)
- 입력값 검증 (SQL Injection / XSS / 비정상 파일)
- 정보 노출 (스택 트레이스 / 민감정보)

## 5. 시나리오 테스트 서버 정보 요청
통합/보안 테스트 완료 후 오케스트레이터에게 서버 정보를 요청한다.

```
## 시나리오 테스트 서버 정보 요청

통합 테스트 및 보안 테스트가 완료됐습니다.
시나리오 테스트 진행을 위해 아래 정보를 제공해주세요.

필요 정보:
- TEST_SERVER_URL: 테스트 서버 URL
- TEST_ADMIN_ACCOUNT: 관리자 계정 (ID/PW)
- TEST_USER_ACCOUNT: 일반 사용자 계정 (ID/PW)
- 기타 필요한 환경 정보

정보 수신 후 시나리오 테스트를 자동으로 시작합니다.
```

## 6. 시나리오 테스트 (서버 정보 수신 후 자동 시작)
유저 스토리 기반 전체 E2E 시나리오를 실제 서버에서 수행한다.

각 시나리오는 TS-001 형식으로 기록:
```
시나리오 ID / 시나리오명 / 전제조건 / 테스트 단계 / 기대결과 / 실제결과
```

## 7. 버그 리포트 작성
모든 테스트 완료 후 `docs/테스트/버그리포트.md` 를 작성한다.

```
BUG-001 형식:
심각도 / 발견단계 / 제목 / 재현경로 / 기대결과 / 실제결과 / 담당 에이전트
```

**Critical / High 버그 발견 시 즉시 중단 후 오케스트레이터 에스컬레이션**

## 8. 버그 수정 후 재테스트
버그 수정 완료 보고 수신 시 해당 케이스 재테스트 + 회귀 테스트를 수행한다.

## 9. 문서 자동 인계
Critical / High 버그 0건 확인 후 문서 에이전트에게 자동 인계한다.

```
## QA 완료 보고
- 통합 테스트: {N}건 PASS
- 보안 테스트: {N}건 PASS
- 시나리오 테스트: {N}건 PASS
- Critical 버그: 0건
- High 버그: 0건
- Medium/Low 버그: {N}건 (docs/테스트/버그리포트.md 참조)
→ /문서시작 자동 실행
```
