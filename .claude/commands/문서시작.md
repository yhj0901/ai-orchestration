# 문서 시작

> 이 커맨드는 QA 에이전트 완료 후 자동 실행되거나
> 오케스트레이터가 직접 호출할 수 있다.

아래 순서로 진행한다.

## 1. 에이전트 하네스 로드
`agents/docs.md` 파일을 읽고 문서 에이전트 규칙을 완전히 숙지한다.

## 2. 입력 확인
```
$ARGUMENTS
```

입력이 없는 경우 아래 산출물을 자동으로 확인한다.

```
확인 목록:
[ ] docs/기획/        ← 기획 에이전트 산출물
[ ] docs/설계/        ← 설계 에이전트 산출물
[ ] docs/테스트/      ← QA 에이전트 산출물
[ ] 프론트엔드 코드
[ ] 백엔드 코드
```

누락 산출물 발견 시 즉시 오케스트레이터에게 보고 후 대기한다.

## 3. 기술 문서 + 사용자 가이드 (병렬 자동 작성)

### 기술 문서 (`docs/기술문서/`)
- `architecture-guide.md` ← architecture.md 기반
- `api-reference.md` ← openapi.yaml 기반 (엔드포인트별 예시 포함)
- `db-guide.md` ← db-schema.md 기반
- `security-guide.md` ← security-design.md 기반
- `dev-setup.md` ← 구현 코드 기반 (로컬 실행 방법)
- `project-structure.md` ← Clean Architecture 레이어 설명

### 사용자 가이드 (`docs/사용자가이드/`)
- `{기능명}-guide.md` ← 유저 스토리 기반 (기능별)
- `scenarios.md` ← 시나리오 테스트 결과 기반
- `faq.md` ← 버그리포트 및 엣지케이스 기반
- `error-guide.md` ← API 에러 코드 기반

## 4. 릴리즈 노트 작성 (`docs/릴리즈노트/v{버전}.md`)

```markdown
# v{버전} 릴리즈 노트
릴리즈 날짜 / 이전 버전

## ⚠️ Breaking Changes
## 🔒 보안 패치
## ✨ 신규 기능 (유저 스토리 ID 기반)
## 🐛 버그 수정 (버그 ID 기반)
## 🔧 기술적 변경사항
## 📋 알려진 이슈
```

## 5. 문서 검토관 최종 검토

```
누락 검토: 모든 기능/엔드포인트/버그수정 반영 여부
일관성 검토: 용어/API명/버전 통일 여부
정확성 검토: 구현과 일치 여부 / 임의 추가 내용 없음
```

검토 실패 항목 발견 시 해당 문서 수정 후 재검토한다.

## 6. 전체 파이프라인 완료 보고
모든 문서 작성 및 검토 완료 후 오케스트레이터에게 최종 보고한다.

```
## 🎉 전체 파이프라인 완료 보고

├── 테스트/       {N}개 파일
├── 기술문서/     {N}개 파일
├── 사용자가이드/ {N}개 파일
```
