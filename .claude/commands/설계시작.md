# 설계 시작

## 사용 예시
```
# 기본 (기획문서 자동 로드)
/설계시작

# 특정 기획문서 지정
/설계시작 docs/기획/로그인.md

# 추가 제약사항 포함
/설계시작 docs/기획/로그인.md / 추가: Redis 세션 캐시 사용, PostgreSQL 필수

# 여러 기획문서 통합 설계
/설계시작 docs/기획/로그인.md, docs/기획/회원가입.md
```

> 이 커맨드는 오케스트레이터의 GATE 1 승인 후에만 실행한다.
> "기획 승인" 또는 "설계 시작해" 명령 없이 실행된 경우 GATE 1 승인 여부를 먼저 확인한다.

---

아래 순서로 진행한다.

## 1. 에이전트 하네스 로드
`agents/design.md` 파일을 읽고 설계 에이전트 규칙을 완전히 숙지한다.

## 2. 입력 산출물 확인
```
$ARGUMENTS
```

입력이 없는 경우 아래를 자동으로 확인한다.
- `docs/기획/` 폴더에서 최신 기획문서 로드
- 기획문서가 없는 경우 오케스트레이터에게 경로 요청

## 3. 기획문서 분석
기획문서에서 아래 항목을 파악한다.
- 유저 스토리 전체 목록
- 기능 요구사항 및 비기능 요구사항
- 보안 요구사항 (CWE 항목)
- 기술 제약사항

## 4. 설계 에이전트 시작
`agents/design.md` 규칙에 따라 6개 서브 에이전트 토론을 시작한다.

확정 기술 스택:
- 아키텍처: Clean Architecture
- 프론트엔드: Next.js (App Router)
- 백엔드: FastAPI
- API 문서: OpenAPI 3.1

토론 순서:
1. 시스템 아키텍트 → 전체 구조 초안
2. API 설계자 + DB 설계자 → 상세 설계
3. 프론트 대변인 + 백엔드 대변인 + 보안 검토관 → 교차 검토
4. 수정 및 조율 → 최종 합의

## 5. 산출물 저장
합의 완료 후 아래 파일을 저장한다.
```
docs/설계/architecture.md
docs/설계/api-spec.md
docs/설계/openapi.yaml
docs/설계/db-schema.md
docs/설계/security-design.md
```

## 6. 완료 보고
```

- 산출물: docs/설계/ 전체
- API 엔드포인트 수: {N}개
- DB 테이블 수: {N}개
- 주요 설계 결정사항: {요약}
- 미결 사항: {있을 경우}
- GATE 2 승인 요청: "설계 승인" 또는 "코드 생성 시작해" 명령을 입력해주세요
```
