# 백엔드 시작

## 사용 예시
```
# 기본 (설계문서 자동 로드)
/백엔드시작

# 설계 승인 명시
/백엔드시작 설계승인됨

# 특정 도메인만 구현
/백엔드시작 인증 도메인만 먼저 구현

# 기존 코드 버그 수정
/백엔드시작 [버그] 주문 조회 API 페이지네이션 누락 문제

# 기존 코드 고도화
/백엔드시작 [고도화] 상품 목록 쿼리 N+1 문제 해결

# 기존 코드 기능 변경 (API 스펙 변경 포함)
/백엔드시작 [변경] 결제 완료 응답에 영수증 URL 필드 추가 (고객 요청)

# DB 스키마 변경 포함
/백엔드시작 [변경] 주문 테이블 status Enum 값 추가 / DB 변경 있음
```

> 이 커맨드는 오케스트레이터의 GATE 2 승인 후에만 실행한다.
> "설계 승인" 또는 "코드 생성 시작해" 명령 없이 실행된 경우 GATE 2 승인 여부를 먼저 확인한다.

---

아래 순서로 진행한다.

## 1. 에이전트 하네스 로드
`agents/backend.md` 파일을 읽고 백엔드 에이전트 규칙을 완전히 숙지한다.

## 2. 입력 확인
```
$ARGUMENTS
```

입력이 없는 경우 아래를 자동으로 로드한다.
- `docs/설계/openapi.yaml` ← 구현 기준 원본
- `docs/설계/db-schema.md` ← DB 구조
- `docs/설계/security-design.md` ← 인증/인가 설계
- `docs/기획/` ← 유저 스토리 확인

## 3. 설계 문서 분석
- openapi.yaml 전체 엔드포인트 및 Pydantic 스키마 매핑
- DB 테이블 구조 및 관계 파악
- 인증/인가 방식 확인
- Clean Architecture 레이어별 구현 목록 도출

## 4. 기능 단위 분리 계획 수립
유저 스토리 기반으로 feature 브랜치 목록을 작성하고 오케스트레이터에게 보고한다.

```
## 백엔드 기능 단위 분리 계획
| 순서 | 브랜치명 | 구현 내용 | DB 변경 | 의존성 |
|------|---------|---------|-------|-------|
| 1 | feature/{명} | {내용} | 있음/없음 | 없음 |
| 2 | feature/{명} | {내용} | 있음/없음 | 1번 완료 후 |

진행 방법: 각 단위 완료 시 코드 리뷰 요청 후 develop 머지
승인 요청: 위 계획대로 진행할까요?
```

## 5. 기능 단위별 구현 (Git Flow 자동 반복)
오케스트레이터 확인 후 아래 사이클을 반복한다.

```
[각 기능 단위마다]
① feature 브랜치 생성
② 도메인 설계 (Entities + Interfaces + 예외)
③ Router + UseCase 구현 + DB 모델 + Repository (병렬)
④ 아키텍처 감시자 레이어 위반 검사
⑤ /openapi.json vs openapi.yaml 일치 검증
⑥ pytest 단위 테스트 + 에러 케이스 작성
⑦ Ruff / mypy --strict / bandit PASS
⑧ 코드 리뷰 요청 (오케스트레이터 응답 대기)
⑨ "머지해" 수신 → develop 머지
⑩ 다음 단위로 이동
```

## 6. 최종 품질 검사
전체 구현 완료 후 전체 코드 대상 최종 검사를 수행한다.
```
Ruff Lint / Ruff Format / mypy --strict / bandit 고위험 0건 / print() 0건
```

## 7. QA 자동 인계
품질 검사 완료 후 프론트 에이전트 완료 여부를 확인 후 QA 에이전트에게 자동 인계한다.

```
## 백엔드 구현 완료 보고
- 구현 완료 엔드포인트: {N}개
- openapi.yaml 일치: 검증 완료
- 테스트 통과: {N}건 / 커버리지: {N}%
- 코드 품질: 전체 PASS
- 미결 사항: {있을 경우}
→ 프론트 완료 확인 후 /QA시작 자동 실행
```

# 백엔드 수정 시작 (기존 코드)

$ARGUMENTS 에 아래 중 하나가 포함된 경우 기존 코드 수정 에이전트로 실행한다.
- `[버그]`, `[고도화]`, `[변경]` 키워드
- 특정 파일/라우터/유스케이스 이름 언급

수정 에이전트 실행 시 `agents/backend.md` 의 **기존 코드 수정 에이전트** 섹션을 따른다.

```
① 코드 분석 → ② 영향도 분석 (DB 영향 포함) → ③ 수정 계획 보고
→ 오케스트레이터 확인 (리스크 High / DB 변경 시 필수)
→ ④ 구현 (DB 변경 시 Alembic + 롤백 스크립트 포함)
→ ⑤ 회귀 테스트 + DB 정합성 확인 → ⑥ 코드 리뷰 → 머지
```
